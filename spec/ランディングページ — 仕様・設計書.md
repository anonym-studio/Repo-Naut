## 1. 概要・目的

| 項目 | 内容 |
|---|---|
| 対象 | **Repo-Naut** 公開用ランディングページ（LP） |
| ホスティング | **Cloudflare Pages**（静的サイト） |
| 役割 | 製品紹介・機能概要・ダウンロード / GitHub へのリンク導線 |
| スコープ外 | Tauri デスクトップアプリ本体の UI・バックエンド・データ永続化 |

LP は **Web ブラウザ向けの静的フロントエンド** のみを扱う。アプリのインストール・実行はデスクトップビルド成果物（`docs/build-guide.md` 参照）側で行う。

## 2. ターゲット・訴求ポイント

### 2-1. ターゲットユーザー

| タイプ | 具体像 |
|---|---|
| マルチプロジェクト勢 | 複数クライアント・複数 OSS を同時並行するフリーランスエンジニア |
| リポジトリ整理勢 | ローカルに溜まった過去のコード資産を棚卸ししたい開発者 |
| CLI パワーユーザー | `gh` CLI / エディタ CLI を使いこなし、さらに効率を上げたい開発者 |

### 2-2. ペインポイント（伝えたい課題）

- Finder / Explorer でリポジトリを探し回る時間の無駄
- `git status` / `git log` を複数ウィンドウで実行する手間
- プロジェクト横断でタスクを管理できるツールがない
- 古くなったリポジトリが邪魔で整理できていない

### 2-3. 訴求ポイント（解決策として伝えること）

| # | メッセージ | 機能的裏付け |
|---|---|---|
| 1 | **「探す」時間を「作る」時間へ** | ワークスペーススキャンで全リポを一覧化。起動時自動スキャン + ファイル監視 |
| 2 | **リポジトリ中心のタスク管理** | カンバンをリポジトリに紐付け。既存ツールとの差別化ポイント |
| 3 | **エディタ・ターミナルへ即アクセス** | カードから VS Code / Cursor / Zed 等を直接起動。ターミナルも同様 |
| 4 | **カードから離れずに完結** | GitHub リンク・エディタ・ターミナル・README プレビュー・カスタムスクリプト実行がカード上に集約 |
| 5 | **Rust 製の軽さと速さ** | Tauri v2 + Rust。メモリ消費が最小限、起動が高速 |
| 6 | **ローカルファースト** | データは JSON ファイルのみ。クラウド不要・プライベート |
| 7 | **キーボードで素早く移動** | `Cmd/Ctrl+K` でコマンドパレット、`g+キー` で Vim 風ナビゲーション |

## 3. コンセプト・ブランドトーン

### 3-1. コンセプトワード

> 「コックピット」— 煩雑になりがちなリポジトリの宇宙（ワークスペース）を、自在に航行し管理・整理するための操縦席。

- **Navigate（航行）**: 迷わず目的のリポジトリへ
- **Organize（整理）**: タグ・アーカイブ・カンバンで体系化
- **Accelerate（加速）**: エディタ起動・スクリプト実行を最短手順で

### 3-2. タグライン

```
Your Repos. Your Mission.
（あなたのリポジトリを、あなたのミッションに。）
```

サブコピー候補:

```
すべてのリポジトリを、一つのコックピットへ。
```

### 3-3. ブランドトーン

- **モダン・テック感**: 宇宙・サイバー・精密機器のイメージ
- **ダークテーマ優先**: 開発者の目に優しく、洗練された印象
- **簡潔**: 機能説明はシンプルな短文。長文ボディコピーは不要

### Section 2: The Problem（Pain Point）

**目的**: ターゲットの共感を得る。課題を具体的に描写する。

コピー案:

```
プロジェクトが増えるほど、リポジトリは迷子になる。

・どのディレクトリにどのプロジェクトがあるか分からない
・git status するためにターミナルを何枚も開く
・「あのコード、どこのリポだっけ」で検索に時間を取られる

```

### Section 3: Core Features（The Solution）

**目的**: 主要機能を 4〜6 点のカードで視覚的に伝える。

| # | 見出し | 本文（案） | 対応機能 |
|---|---|---|---|
| 1 | ワークスペースを一瞬でスキャン | 登録したフォルダ配下の全リポジトリを Rust の並列処理で高速検索。最新コミット・ブランチ・未コミット数をリアルタイム表示。 | `scan_workspace` / `rayon` / `notify` |
| 2 | カードがコックピットになる | GitHub/GitLab/Bitbucket リンク・エディタ起動・ターミナル起動・カスタムスクリプト実行・README プレビューがカード上に集約。VS Code / Cursor / Zed 等を登録して即起動。 | `RepoCard` / `EditorButton` / `ScriptRunButton` / `ReadmeModal` |
| 3 | リポジトリ横断カンバン | Todo / In Progress / Review / Done の 4 カラム。タスクはリポジトリに紐付けられる、開発者のためのタスク管理。 | `Kanban` / `@dnd-kit` |
| 4 | アーカイブで整理 | 古いプロジェクトは `_archive/` へ移動。スナップショットメタを保存し、必要なときに元のパスへ復元できる。 | `archive_repo` / `restore_repo` |
| 5 | GitHub PAT 連携 | PAT を OS キーチェーンに安全保管。PR 数・Issue 数をカードにリアルタイム表示。 | `github.rs` / `keyring` |
| 6 | キーボードで即アクセス | `Cmd/Ctrl+K` でコマンドパレット（リポジトリ・タスク・コミットを横断検索）。`g+キー` の Vim 風ナビゲーション、`?` でショートカット一覧。 | `CommandPalette` / `ShortcutsHelp` |

### Section 4: Technology（なぜ速いのか）

**目的**: 技術的な信頼感を与える。エンジニア読者向け。

- **Tauri v2 + Rust** — Electron の数分の一のメモリ消費。ネイティブバイナリで OS に溶け込む
- **`git2`（libgit2）** — Git 操作を Rust から直接呼び出し。コマンドプロセスを起動しないため高速
- **`rayon`** — ワークスペーススキャンを並列処理。リポジトリが 100 件超えても遅くならない
- **`notify` + debounce** — ファイルシステムを監視し、`.git/HEAD` の変化を 800ms debounce で検知。コミット後に自動再スキャン
- **ローカルファースト** — データは JSON 3 ファイルのみ（`settings.json` / `repos-meta.json` / `kanban.json`）。クラウドなし・会員登録なし

技術スタック表示（バッジ / アイコン列）:
```
Rust  ·  Tauri v2  ·  React 18  ·  TypeScript  ·  Tailwind CSS
```

## 7. 技術スタック

デスクトップアプリと **思想を揃えつつ依存は分離** する。

| レイヤ | 選定 | 理由 |
|---|---|---|
| ビルド | Vite 6 | ルートアプリと同系。高速な dev / build |
| UI | React 18 | 既存スキル・コンポーネント資産の流用可能性 |
| スタイル | Tailwind CSS v4（`@import "tailwindcss"`） | ルートと同じ v4 方式。`tailwind.config.js` 不要 |
| 言語 | TypeScript（strict） | ルートと同水準の型安全 |
| ルーティング | 当面なし（単一ページ） | 将来複数ページ化する場合は `react-router-dom` + `BrowserRouter` |
| ホスティング | Cloudflare Pages | Git 連携・プレビューデプロイ・CDN・無料枠 |

**意図的に含めないもの:**

| パッケージ | 理由 |
|---|---|
| `@tauri-apps/*` | Web では利用不可 |
| `@tanstack/react-query` / `zustand` | LP 初期段階ではサーバーデータ・複雑 UI ステートが不要 |
| `createHashRouter` | Tauri 専用。LP は通常 URL（`BrowserRouter`）を使う |

## 10. デスクトップアプリとの分離ルール

| # | ルール | 理由 |
|---|---|---|
| 1 | LP のコードは `landing/` 以下にのみ置く | ルート `pnpm build` が Tauri 用 `dist/` を壊さないため |
| 2 | LP から `invoke()` / Tauri API を呼ばない | Web 実行環境に存在しない |
| 3 | LP の型・Hook は `landing/src/` 内で完結させる | `src/types/` や `src/hooks/` はデスクトップ専用 |
| 4 | ルーティングは `BrowserRouter`（導入時） | Cloudflare Pages ではクリーン URL が可能 |
| 5 | ダウンロードリンクは外部 URL | GitHub Releases 等。バイナリを Pages に載せない |

## 11. ローカル開発

### 11-1. 初回セットアップ

```bash
# リポジトリルートで実行（workspace 全体をインストール）
pnpm install
```

### 11-2. 開発サーバー

```bash
# ルートから
pnpm landing:dev

# または landing ディレクトリ直下で
cd landing && pnpm dev
```

- URL: `http://localhost:1430`
- デスクトップアプリ（`pnpm tauri:dev`、ポート 1420）と **同時起動可能**

### 11-3. ビルド・型チェック

```bash
pnpm landing:type-check      # TypeScript チェック
pnpm landing:build           # 本番ビルド → landing/dist/
pnpm landing:preview         # ビルド成果物のローカル確認
```

### 11-4. コミット前の検証

LP のみ変更した場合:

```bash
pnpm landing:type-check && pnpm landing:build
```

デスクトップと LP の両方を変更した場合:

```bash
pnpm type-check && cargo check --manifest-path src-tauri/Cargo.toml
pnpm landing:type-check && pnpm landing:build
```

### 11-5. ルート `package.json` に追加するスクリプト

| スクリプト | コマンド |
|---|---|
| `landing:dev` | `pnpm --filter repo-naut-landing dev` |
| `landing:build` | `pnpm --filter repo-naut-landing build` |
| `landing:preview` | `pnpm --filter repo-naut-landing preview` |
| `landing:type-check` | `pnpm --filter repo-naut-landing type-check` |

## 12. Cloudflare Pages デプロイ

### 12-1. 方針

- **Cloudflare Pages** で公開: **https://repo-naut.pages.dev/**
- **Workers Builds / `*.workers.dev` は使わない**（静的 LP には Pages が適切）
- **Git 連携デプロイ**（main ブランチ → 本番、PR → プレビュー URL 自動生成）
- ビルドは Cloudflare 側で実行。`landing/dist/` を配信
- カスタムドメインは Cloudflare ダッシュボードで後から設定

詳細手順・Workers からの移行: [`docs/cloudflare-pages-deploy.md`](../docs/cloudflare-pages-deploy.md)

### 12-2. ダッシュボード設定（初回）

**Create → Pages → Connect to Git**（Workers タブや「Import to Workers」ではない）。

| 項目 | 値 |
|---|---|
| リポジトリ | 本リポジトリ（`Repo-Naut`） |
| **Root directory** | `landing` |
| **Build command** | `pnpm install && pnpm run build` |
| **Build output directory** | `dist` |
| **Framework preset** | None（Vite を自前ビルド） |

### 12-3. 環境変数

| 変数名 | 値（例） | 用途 |
|---|---|---|
| `NODE_VERSION` | `20` | Node.js バージョン固定 |
| `PNPM_VERSION` | `10` | ルート `packageManager` と整合 |

ビルドコマンド（corepack 経由の場合）:

```bash
corepack enable && pnpm install && pnpm run build
```

### 12-4. SPA ルーティング

現状 LP はアンカーリンク（`#download` 等）のみで **`react-router-dom` 未使用**のため、`_redirects` は不要。

将来クライアントルートを導入する場合:

| ホスティング | 設定 |
|---|---|
| **Cloudflare Pages（静的）** | `landing/public/_redirects` に `/* /index.html 200` |
| **Workers（static assets）** | `_redirects` の `/* → /index.html` はデプロイエラーになるため、`wrangler.jsonc` の `assets.not_found_handling = "single-page-application"` を使用 |

### 12-5. デプロイフロー

```
GitHub main ブランチへの push
  → Cloudflare Pages が自動ビルド
  → https://repo-naut.pages.dev/（本番）に反映
  → カスタムドメイン（任意）を Pages プロジェクトに紐付け

Pull Request
  → プレビュー URL を自動生成
  → レビュー・確認後にマージ
```

### 12-6. ダウンロード導線

LP 上の「ダウンロード」ボタンは **GitHub Releases**（または将来の配布 URL）への外部リンクとする。バイナリ（`.dmg` / `.msi` / `.exe`）は Cloudflare Pages には配置しない。
